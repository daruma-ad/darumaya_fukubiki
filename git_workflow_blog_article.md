# 複数PCでのGit運用術：安全に開発を引き継ぐ「3つの鉄則」とブランチの活用法

職場と自宅、メインPCとサブPCなど、複数のパソコンで1つのプロジェクト（リポジトリ）を開発する際、「あっちのPCでの変更が反映されていない」「ファイルが競合（Conflict）してしまった」といったトラブルに悩まされていませんか？

この記事では、**複数PC間で安全かつスムーズに開発を引き継ぐための「3つの鉄則」**と、現場でよく使われる**「ブランチ（枝）」を活用したワークフロー**について分かりやすく解説します。

---

## 複数PCで開発する時の「3つの鉄則」

トラブルを防ぐために、まずは毎日の作業サイクルを以下のルールで固定しましょう。

### 1. 開発を「始める前」に、必ず `git pull` する（超重要！）
もう一方のPCで進めた変更を取り逃がさないために、作業開始時のルーティンにしましょう。
これを忘れてコードを書き始めると、後で競合が起きて修正が非常に面倒になります。

```bash
# ターミナルを開き、プロジェクトのフォルダで実行
git pull origin main
```

### 2. 作業が一段落したら `git push` する
コードが動く状態（テストが通る、画面が崩れていないなど）になったら、GitHubに反映させます。
**「一度動く状態」でPushする**のが、引き継ぎをスムーズにするコツです。エラーが出たまま投げっぱなしにしないようにしましょう。

```bash
git add .
git commit -m "本日の作業内容を保存（〇〇の機能を追加）"
git push origin main
```

### 3. 本格的な開発では「ブランチ（Branch）」を活用する
最初は `main` ブランチに直接プッシュしても問題ありませんが、機能追加やバグ修正など、ある程度のまとまった作業を行う場合は「ブランチ」を分けるのがベストプラクティスです。
ブランチとは、「現在のコードに変更を加えず、安全に新しい作業をするための専用のコピー（枝）」のことです。

---

## 実践：ブランチを使った開発サイクル

ブランチを使った開発は、以下の **「5つのステップ」** が基本となります。

### ステップ1：新しいブランチを作る ＆ 切り替える（PC-A）
まずは `main`（本番）から枝分かれした作業用ブランチを作ります。

```bash
git checkout -b feature/login-page
```
*   `feature/login-page` はブランチの名前です。
*   `-b` オプションをつけることで、作成と同時にそのブランチへ自動的に**切り替わり（移動）**ます。

> 💡 **名前の付け方のコツ**
> *   機能追加： `feature/〇〇`
> *   バグ修正： `bugfix/〇〇`
> このように接頭辞をつけると、何のためのブランチか一目で分かりやすくなります。

### ステップ2：開発 ＆ コミットする（PC-A）
ファイルを編集したら、いつも通りコミットします。
ここでコミットされた内容は、**作成したブランチにだけ記録され、元の `main` には一切影響を与えません。**

```bash
git add .
git commit -m "ログイン画面のレイアウトを作成"
```

### ステップ3：GitHubにプッシュする（PC-A）
作成したブランチを、GitHubにアップロード（保存）します。

```bash
# ブランチ名を指定してPush
git push origin feature/login-page
```
これで、クラウド上のGitHubに安全な作業の「枝」が保存されました。

### ステップ4：別のPCで続きをやる（PC-Bへ移動）
別のPCに移動したら、まずはGitHubに上がった新しいブランチの情報を取得し、そのブランチに切り替えて作業を開始します。

```bash
# 1. GitHubの最新情報を読み込む（fetch）
git fetch origin

# 2. そのブランチに切り替える
git checkout feature/login-page
```
これでもう一方のPCで作った「枝」の続きから、ローカル環境で開発を再開できます。
作業が終われば、また「ステップ2」と「ステップ3」を繰り返します。

### ステップ5：開発が終わったら統合（Merge）する
機能のテストも終わり、すべて完璧に動く状態になったら、最後に大元の `main` ブランチに統合（マージ）します。

```bash
# 1. まず main ブランチに戻る
git checkout main

# 2. 前提として、統合前に main を最新にしておくのが安全です
git pull origin main

# 3. 作ったブランチを main にマージ（合流）する
git merge feature/login-page

# 4. 統合された最新の main を GitHub に反映！
git push origin main
```

---

## まとめ：日常のサイクル

このブログ記事で紹介した手順を、日常のサイクルとして体に染み込ませましょう。

1.  **PC-Aで作業**: `pull` ➡ ブランチを切る ➡ 開発 ➡ `commit` ➡ ブランチを `push`
2.  **PC-Bで作業**: `fetch` ➡ ブランチをチェックアウト ➡ 開発 ➡ `commit` ➡ `push`
3.  **完成したら**: `main` に切り替え ➡ マージ ➡ `main` を `push`

このリズムを意識すれば、複数PCでの開発はもちろん、チーム開発にもスムーズに移行することができます。ぜひ試してみてください！
