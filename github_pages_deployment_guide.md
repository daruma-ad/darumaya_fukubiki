# GitHub Pagesでサイトを公開！「Deploy from a branch」と「GitHub Actions」どちらを選ぶべき？

GitHubに作成したHTMLファイルなどを、無料でWebサイトとして公開できる便利な機能「**GitHub Pages**」。

いざ設定画面を開くと、**「Source（ソース）」**という項目で2つのデプロイ（公開）方法を選ぶように求められます。

1.  **Deploy from a branch**
2.  **GitHub Actions**

「どっちを選べばいいの？」と迷う方も多いのではないでしょうか。
この記事では、この2つの方法の違いと、それぞれの「おすすめなケース」を分かりやすく解説します！

---

## 1. Deploy from a branch（ブランチからのデプロイ）

**▶︎ 一言でいうと：「そのままポン置き」**

指定したブランチ（通常は `main` や `master`）の中に入っているファイルを、**そのままの形**でサーバーに配置して公開する方法です。

### 📌 おすすめなケース
*   **HTML、CSS、生のJavaScript（Vanilla JS）だけで作られたサイト**
*   **LP（ランディングページ）や、シンプルなポートフォリオサイト**
*   とりあえず手軽に、最速で公開したいとき

### ⭕️ メリット
*   **とにかく簡単！**：ドロップダウンで対象のブランチを選ぶだけで、数分後にはサイトが公開されます。
*   特別な設定ファイル（YAMLファイルなど）を書く必要がありません。

### ❌ デメリット
*   「ビルド」という作業（プログラムの変換作業）ができません。そのため、ReactやVueなどのモダンなフレームワークを使ったアプリには不向きです。

---

## 2. GitHub Actionsを使ったデプロイ

**▶︎ 一言でいうと：「ロボットが作業をしてから公開」**

GitHubの自動化機能である「GitHub Actions」を使って、サーバーに公開するまでの過程（プログラムの変換、テストの実行など）を自動で行わせる方法です。

### 📌 おすすめなケース
*   **React、Next.js、Vue、Nuxt.jsなどのフレームワークを使っている場合**
*   **Sass（SCSS）を使っていたり、TypeScriptをJavaScriptに変換する必要がある場合**
*   **公開する前に自動テスト（Lintなど）を走らせたい場合**
*   GatsbyやJekyllなどの静的サイトジェネレーターを使う場合

### ⭕️ メリット
*   **超高機能**：ただ公開するだけでなく、「コードの変換（ビルド）」→「テスト」→「公開」という複雑な連係プレイを全自動化できます。
*   公式やコミュニティが用意した便利なテンプレートが豊富にあるため、設定ファイルをコピペするだけで高度な公開パイプラインが作れます。

### ❌ デメリット
*   設定ファイル（`.github/workflows/xxx.yml`）の書き方を少し勉強する必要があります。
*   「そのままポン置き」に比べると、公開までに数分のビルド時間がかかることがあります。

---

## 💡 まとめ：結局どっちを選べばいい？

迷ったときは、以下を基準に判断しましょう！

*   あなたが作っているのが**「`.html` ファイルをブラウザで開いて直接見れるサイト」**なら
    👉 **『Deploy from a branch』 を選ぼう！**
    設定が最も簡単です。

*   あなたが作っているのが**「`npm run build` などのコマンドを実行しないと見れないサイト」**なら
    👉 **『GitHub Actions』 を選ぼう！**
    ビルド作業をロボットに任せることができます。

あなたのプロジェクトに合ったデプロイ方法を選んで、快適なWeb制作ライフを送りましょう！
